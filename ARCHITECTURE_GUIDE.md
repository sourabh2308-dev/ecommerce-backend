/api/auth/login` - Validate credentials, issue token pair\n  - `POST /api/auth/refresh` - Rotate tokens (old revoked, new issued)\n  - `POST /api/auth/logout` - Revoke refresh token\n- **Database**: auth_db (RefreshToken storage for revocation)\n- **Dependencies**: user-service (fetch user data via internal call)\n- **Security**: BCrypt password hashing, HS256 JWT signing\n- **Important Classes**:\n  - `JwtUtil`: Token generation/validation with JJWT library\n  - `AuthServiceImpl`: Business logic for login/refresh/logout\n  - `AuthController`: REST endpoints\n\n#### 2. **User Service** (Port 8081)\n- **Purpose**: User profile management and seller verification\n- **Key Operations**:\n  - `POST /api/user/register` - Create user account\n  - `GET /api/user/{uuid}` - Fetch user profile\n  - `POST /api/user/seller-details` - Submit seller verification docs\n  - `PUT /api/user/profile` - Update profile\n  - `POST /api/user/verify-otp` - Verify email via OTP\n- **Database**: user_db (User, OTPVerification, SellerDetail tables)\n- **States**: \n  - ACTIVE: Email verified, can place orders\n  - PENDING_DETAILS: Seller submitted incomplete info\n  - PENDING_APPROVAL: Seller awaiting admin verification\n- **Important Classes**:\n  - `User`: Main user entity with email, role, status\n  - `SellerDetail`: Additional seller info (business name, documents)\n  - `OTPVerification`: Email verification tokens\n  - `UserServiceImpl`: Handles registration, OTP, profile updates\n\n#### 3. **Product Service** (Port 8081)\n- **Purpose**: Product catalog management\n- **Key Operations**:\n  - `POST /api/product` - Create product (seller)\n  - `GET /api/product/{uuid}` - Fetch product details\n  - `GET /api/product` - Search/paginate products\n  - `PUT /api/product/{uuid}` - Update product (seller)\n  - `DELETE /api/product/{uuid}` - Soft-delete product\n- **Database**: product_db (Product table)\n- **Caching**: Redis (product details, search results)\n- **Features**:\n  - Multi-image support (semicolon-separated URLs)\n  - Stock management (created by order-service via Feign)\n  - Category filtering\n- **Important Classes**:\n  - `Product`: Entity with price, stock, seller, status\n  - `ProductServiceImpl`: CRUD, search, stock reduction\n  - `ProductController`: API endpoints\n\n#### 4. **Order Service** (Port 8081)\n- **Purpose**: Order management and fulfillment\n- **Key Operations**:\n  - `POST /api/order` - Create order (buyer)\n  - `GET /api/order/{uuid}` - Fetch order details\n  - `PUT /api/order/{uuid}/status` - Update status (admin)\n  - `GET /api/order` - List buyer's orders (paginated)\n- **Database**: order_db (Order, OrderItem tables)\n- **Event Flow**:\n  - Creates order → publishes `order.created` Kafka event\n  - Consumes `payment.completed` → updates paymentStatus\n  - Saga pattern: Payment failure → cancels order, restores stock\n- **Order Status**: CREATED → CONFIRMED → SHIPPED → DELIVERED\n- **Payment Status**: PENDING → SUCCESS/FAILED\n- **Important Classes**:\n  - `Order`: Main order entity with items, total, status\n  - `OrderItem`: Individual item in order (qty, price, seller)\n  - `PaymentEventConsumer`: Listens to payment events, updates order\n  - `OrderServiceImpl`: CRUD, status transitions, payment saga\n\n#### 5. **Payment Service** (Port 8081)\n- **Purpose**: Payment processing and settlement\n- **Key Operations**:\n  - `POST /api/payment` - Process payment (buyer)\n  - `GET /api/payment` - Fetch payment history (buyer)\n  - `GET /api/seller/dashboard` - Seller payment dashboard\n  - `GET /api/admin/dashboard` - Admin finance dashboard\n- **Database**: payment_db (Payment, PaymentSplit tables)\n- **Event Flow**:\n  - Consumes `order.created` Kafka event\n  - Simulates payment (100% success for demo)\n  - Publishes `payment.completed` event → triggers order status update\n- **Features**:\n  - Split payments (multi-seller orders)\n  - Platform commission calculation (10% default)\n  - Delivery fees per item (₹30 default)\n- **Important Classes**:\n  - `Payment`: Main payment entity with amount, status, order UUID\n  - `PaymentSplit`: Settlement record per seller\n  - `OrderEventConsumer`: Listens for order.created, initiates payment\n  - `PaymentServiceImpl`: Payment processing, splits, dashboards\n\n#### 6. **Review Service** (Port 8081)\n- **Purpose**: Product reviews and ratings\n- **Key Operations**:\n  - `POST /api/review` - Create review (buyer, post-delivery)\n  - `GET /api/review/product/{uuid}` - Get product reviews\n  - `PUT /api/review/{uuid}` - Update own review\n  - `DELETE /api/review/{uuid}` - Delete review\n  - `GET /api/review/my-reviews` - Get my reviews\n- **Database**: review_db (Review table)\n- **Caching**: Redis (product review aggregate)\n- **Constraints**:\n  - Only buyers who've purchased product can review\n  - Only after order is DELIVERED\n  - One review per buyer per product\n- **Important Classes**:\n  - `Review`: Entity with rating (1-5), comment, timestamp\n  - `ReviewServiceImpl`: CRUD, duplicate check, delivery validation\n  - `ReviewController`: API endpoints\n\n---\n\n## Technology Stack\n\n### Core Framework\n- **Spring Boot 3.3**: Latest version with native compilation support\n- **Spring Cloud**: Service discovery, config management, |\n  ├─→ POST /api/user/register {email, password, role}\n  │   │\n  │   └→ USER-SERVICE\n  │       ├─ Validate email format\n  │       ├─ Hash password with BCrypt\n  │       ├─ Check email not already registered\n  │       ├─ Create User entity (status=PENDING_EMAIL)\n  │       ├─ Generate OTP\n  │       ├─ Send email with OTP\n  │       └─ Return {userId, email, status}\n  │\n  ├─→ POST /api/user/verify-otp {email, otp}\n  │   │\n  │   └→ USER-SERVICE\n  │       ├─ Validate OTP not expired (5 min)\n  │       ├─ Mark email as verified\n  │       ├─ Update user status to ACTIVE\n  │       └─ Return {verified: true}\n  │\n  └─→ POST /api/auth/login {email, password}\n      │\n      └→ AUTH-SERVICE\n          ├─ Call user-service via RestTemplate\n          │  (GET /api/user/internal/email/{email})\n          ├─ Fetch user password hash\n          ├─ Validate password with BCrypt.matches()\n          ├─ Check emailVerified && accountActive\n          ├─ Generate JWT access token (15 min expiry)\n          │  Payload: {sub: email, uuid: userUuid, role: role, iat, exp}\n          ├─ Generate refresh token (UUID, 7 day expiry)\n          ├─ Store refresh token in auth_db\n          └─ Return {accessToken, refreshToken, tokenType: \"Bearer\"}\n\nAPI Gateway (for subsequent requests):\n  ├─ Extract JWT from Authorization: Bearer header\n  ├─ Validate signature with secret key\n  ├─ Check expiry (fail if exp < now)\n  ├─ Extract claims (uuid, role)\n  ├─ Forward request with X-User-Uuid, X-User-Role headers\n  └─ If JWT expired: client calls POST /api/auth/refresh {refreshToken}\n      └─ Auth-Service rotates tokens (revoke old, issue new pair)\n```\n\n### 🛒 Order Creation Flow (Synchronous + Asynchronous Saga)\n\n```\nCLIENT (Buyer)\n  |\n  └─→ POST /api/order {items: [{productUuid, quantity}, ...]}\n       │\n       ├─→ API-GATEWAY (validate JWT, extract X-User-Uuid)\n       │\n       └─→ ORDER-SERVICE\n           ├─ Validate role == \"BUYER\"\n           ├─ For each item:\n           │  ├─ Call product-service (Feign): getProduct(productUuid)\n           │  ├─ Check if product ACTIVE\n           │  ├─ Check if stock >= quantity\n           │  ├─ Call product-service: reduceStock(productUuid, qty)\n           │  └─ Create OrderItem entity\n           │\n           ├─ Calculate totalAmount = sum(price * qty for all items)\n           ├─ Create Order entity\n           │  {uuid, buyerUuid, items, totalAmount, status: CREATED, paymentStatus: PENDING}\n           ├─ Save Order to order_db\n           │\n           ├─ PUBLISH KAFKA EVENT: order.created\n           │  Topic: order.created\n           │  Payload: {orderUuid, buyerUuid, amount, items: [{productUuid, sellerUuid, amount}, ...]}\n           │  Partitions: Ensures order events for same buyer go to same partition (ordered)\n           │\n           └─ RESPOND TO CLIENT: OrderResponse {uuid, items, totalAmount, paymentStatus: \"PENDING\"}\n\n       ▼ ASYNCHRONOUS (Kafka Saga)\n\n       PAYMENT-SERVICE (consumes order.created)\n           ├─ Listen on order.created topic (group: payment-service)\n           ├─ Check if event already processed (idempotency via ProcessedEvent table)\n           ├─ Create Payment entity {uuid, orderUuid, amount, status: INITIATED}\n           ├─ Simulate payment gateway (100% success for demo)\n           ├─ Update Payment {status: SUCCESS} (or FAILED randomly)\n           ├─ Create PaymentSplit entries per seller\n           │  ├─ Seller amount = (orderAmount - fees) * seller_ratio\n           │  ├─ Platform commission = amount * 10%\n           │  ├─ Delivery fees = qty * ₹30\n           │\n           ├─ PUBLISH KAFKA EVENT: payment.completed\n           │  Topic: payment.completed\n           │  Payload: {orderUuid, paymentUuid, status: SUCCESS}\n           │  Partition: Same as order to maintain causality\n           │\n           └─ Mark ProcessedEvent (prevent duplicate processing)\n\n       ORDER-SERVICE (consumes payment.completed)\n           ├─ Listen on payment.completed topic (group: order-service)\n           ├─ Fetch Order by orderUuid\n           ├─ Update Order {paymentStatus: SUCCESS}\n           ├─ If status == FAILED:\n           │  ├─ Update Order {status: CANCELLED, paymentStatus: FAILED}\n           │  ├─ Call product-service: restoreStock(productUuid, qty) for each item\n           │  └─ Saga compensation (rollback)\n           │\n           └─ Save updated Order\n\n       CLIENT (Frontend)\n           ├─ Polling: GET /api/order/{orderUuid}\n           │  Every 2 seconds while paymentStatus == \"PENDING\"\n           │  Stops when status changes to SUCCESS/FAILED\n           │\n           └─ On SUCCESS: Display confirmation, redirect to order details\n```\n\n### 💰 Payment Processing (Multi-Seller Split)\n\n```\nAssume Order with 2 sellers:\n  - Seller A: 1 item @ ₹1000\n  - Seller B: 2 items @ ₹500 each = ₹1000\n  - Total: ₹2000\n\nPayment Calculation:\n  1. Total Amount: ₹2000\n  2. Platform Commission (10%): ₹200\n  3. Delivery Fees (per item, ₹30 each, 3 items): ₹90\n  4. Seller A receives: ₹1000 - (₹200 * 1000/2000) - (₹30 * 1 item) = ₹900\n  5. Seller B receives: ₹1000 - (₹200 * 1000/2000) - (₹30 * 2 items) = ₹840\n  6. Platform keeps: ₹200 + ₹90 - refunds = ₹290\n\nPaymentSplit entities created:\n  - id: 1, paymentUuid, sellerUuid: A, amount: ₹900, status: PENDING\n  - id: 2, paymentUuid, sellerUuid: B, amount: ₹840, status: PENDING\n  - Commission entry, status: SETTLED\n\nSeller Dashboard shows:\n  - Available balance: Sum of PAID PaymentSplit amounts\n  - Pending balance: Sum of PENDING PaymentSplit amounts\n```\n\n### ⭐ Review Creation Flow\n\n```\nCLIENT (Buyer, post-delivery)\n  |\n  └─→ POST /api/review {orderUuid, productUuid, rating, comment}\n       │\n       └─→ REVIEW-SERVICE\n           ├─ Validate role == \"BUYER\"\n           ├─ Call order-service (Feign): getOrder(orderUuid)\n           ├─ Check order.buyerUuid == requester UUID\n           ├─ Check order.status == \"DELIVERED\"\n           ├─ Check productUuid in order.items\n           ├─ Check no existing review by same buyer for same product\n           │  (idempotency: allow update, disallow duplicate)\n           │\n           ├─ Create Review entity\n           │  {uuid, orderUuid, productUuid, sellerUuid, buyerUuid, rating, comment}\n           │\n           ├─ PUBLISH KAFKA EVENT: review.submitted\n           │  Topic: review.submitted\n           │  Payload: {reviewUuid, productUuid, rating, buyerUuid}\n           │\n           ├─ Invalidate product review cache (Redis)\n           │\n           └─ RESPOND: ReviewResponse {uuid, productUuid, rating, comment, createdAt}\n\n       PRODUCT-SERVICE (consumes review.submitted)\n           ├─ Update product review aggregate (cached)\n           │  ├─ Increment review count\n           │  ├─ Recalculate average rating\n           │  └─ Invalidate cache\n           |\n           └─ No direct response needed (async)\n```\n\n---\n\n## Service Details\n\n### API Layers\n\n```\n┌──────────────────────────────────────────────────────────┐\n│ 1. EXTERNAL (Client facing via API Gateway)              │\n│    - JWT Token Validation (HS256 signature check)        │\n│    - Rate Limiting (RateLimitGlobalFilter)              │\n│    - Correlation ID Injection                           │\n├──────────────────────────────────────────────────────────┤\n│ 2. INTERNAL (Service-to-Service)                         │\n│    - X-Internal-Secret Header (InternalSecretFilter)    │\n│    - Feign Client Interceptors                          │\n│    - Circuit Breaker (Resilience4j)                     │\n├──────────────────────────────────────────────────────────┤\n│ 3. METHOD LEVEL                                          │\n│    - @PreAuthorize(\"hasRole('SELLER')\")               │\n│    - Custom @Secured annotations                        │\n│    - Authorization logic in service layer                │\n├──────────────────────────────────────────────────────────┤\n│ 4. DATABASE LEVEL                                        │\n│    - Row-level security via userUuid checks             │\n│    - Soft deletes (isDeleted flag)                      │\n│    - Audit timestamps (createdAt, updatedAt)            │\n└──────────────────────────────────────────────────────────┘\n```\n\n### JWT Token Structure\n\n```\nAccess Token (Header.Payload.Signature):\n{\n  \"alg\": \"HS256\",        // Header\n  \"typ\": \"JWT\"\n}\n{\n  \"sub\": \"user@example.com\",  // Subject (email)\n  \"uuid\": \"abc-123\",           // User unique ID\n  \"role\": \"BUYER\",             // Role for authorization\n  \"iat\": 1708857600,           // Issued at timestamp\n  \"exp\": 1708858500            // Expiration (15 mins later)\n}\nSignature = HMAC-SHA256(header + payload, secret_key)\n\nUsage: Authorization: Bearer {access_token}\nValidation:\n  1. Split on '.' to get 3 parts: header, payload, signature\n  2. Verify signature matches HMAC(header.payload)\n  3. Decode payload, check exp < now\n  4. Extract UUID, role for authorization\n```\n\n---\n\n## Development Guide\n\n### Project Structure\n\n```\necommerce-backend/\n├── auth-service/              # JWT issuance & refresh\n│   ├── src/main/java/\n│   │   └── com/sourabh/auth_service/\n│   │       ├── controller/    # REST endpoints\n│   │       ├── service/       # Business logic\n│   │       ├── entity/        # JPA entities\n│   │       ├── repository/    # Data access\n│   │       ├── security/      # JWT filters\n│   │       ├── config/        # Spring configs\n│   │       ├── util/          # JWT utility\n│   │       └── exception/     # Custom exceptions\n│   └── src/main/resources/\n│       ├── application.properties  # Config from config-server\n│       └── db/migration/           # Flyway migrations\n│\n├── user-service/              # User profiles & seller verification\n│   ├── src/main/java/\n│   │   └── com/sourabh/user_service/\n│   │       ├── controller/\n│   │       ├── service/\n│   │       ├── entity/        # User, SellerDetail, OTP\n│   │       ├── repository/\n│   │       ├── config/\n│   │       ├── security/      # Interceptors\n│   │       └── exception/\n│   └── src/main/resources/\n│\n├── product-service/           # Product catalog\n│   ├── src/main/java/\n│   ├── src/main/resources/\n│   └── src/test/java/         # Unit & integration tests\n│\n├── order-service/             # Orders & fulfillment\n│   ├── src/main/java/\n│   │   └── com/sourabh/order_service/\n│   │       ├── kafka/         # Kafka consumers\n│   │       │   ├── consumer/  # PaymentEventConsumer\n│   │       │   └── event/     # OrderCreatedEvent, OrderItemEvent\n│   │       ├── feign/         # RestClient to product-service\n│   │       ├── entity/        # Order, OrderItem\n│   │       └── ... Payment):\nOption 1: Distributed 2PC (locks, slow) ❌\nOption 2: Saga with Kafka events (async, compensating) ✅\n\nHappy Path:\n  order-service creates Order → publishes order.created\n  payment-service creates Payment → publishes payment.completed\n  order-service updates paymentStatus\n\nCompensating Path (if payment fails):\n  order-service cancels Order\n  order-service calls product-service to restore stock\n  Ensures no orphaned orders or inventory issues\n```\n\n### 2. **Idempotency** (Prevent Duplicate Processing)\n```\nProblem: Kafka message reprocessed due to network retry\nSolution: ProcessedEvent table tracks {eventId, eventType, createdAt}\n\nPayment consumer checks:\n  if (processedEventRepository.existsByEventId(\"payment-completed:{paymentUuid}\"))\n    return;  // Already processed, skip\n  \n// Process new event\nprocess(event);\nprocessedEventRepository.save(new ProcessedEvent(eventId));\n```\n\n### 3. **Circuit Breaker** (Resilience)\n```\nproblem: order-service calls product-service, product-service down\nSolution: Resilience4j circuit breaker\n\nStates:\n  - CLOSED: Normal operation, requests pass through\n  - OPEN: Failures detected, fast-fail without calling service\n  - HALF_OPEN: Try a few requests to see if service recovered\n\nConfig:\n  failureRateThreshold=${resilience4j.circuitbreaker.instances.product-service.failure-rate-threshold}\n  slidingWindowSize=${resilience4j.circuitbreaker.instances.product-service.sliding-window-size}\n```\n\n### 4. **JWT vs RefreshToken**\n```\nAccess Token:\n  - Stateless (no DB lookup)\n  - Short-lived (15 min)\n  - Cannot be revoked (design tradeoff)\n  - Signed with secret key\n\nRefresh Token:\n  - Stateful (stored in DB)\n  - Long-lived (7 days)\n  - Revocable (logout)\n  - Rotation pattern (old revoked when new issued)\n  - Used to get new access token\n```\n\n### 5. **Soft Deletes** (Data Retention)\n```\nInstead of DELETE, set isDeleted=true\nBenefits:\n  - Audit trail (know what was deleted and when)\n  - Referential integrity (foreign keys still valid)\n  - Recovery possible (restore old records)\n\nUsage:\n  SELECT * FROM orders WHERE isDeleted = false\n  (always include this filter in queries)\n```\n\n---\n\n## Common Workflows\n\n### Adding New API Endpoint\n\n1. **Define DTO** (Data Transfer Object)\n   ```java\n   public class CreateProductRequest {\n       @NotBlank\n
